Stories about getting stuck with Google Adsense, site structure, etc.

発行日時: 2022.03.06 05:51

最終更新日: 2022.04.04 05:16

This site is composed of

Name.com - > Route 53 - > AWS ALB - (http1.1)> AWS EC2.

This is the way it is supposed to be. At the time of implementation, the

Name.com - > AWS ALB - (http2.0)> AWS EC2.

but it was registered with adsense & ALB's specs.

There were two points of stumbling blocks in registering with adsense, so I'll leave a note.

1. when I try to register with adsense I get an error that the site is unreachable

As it turns out, [Http Error 464](https://docs.aws.amazon.com/ja_jp/elasticloadbalancing/latest/application/load- balancer- troubleshooting.html #http- 464- issues) was occurring.

It seems that the http being requested did not match the protocol inside the load balancer and was throwing an error.

Communication to the ALB target group needed to be set to http1.1, and nginx on EC2 also needed to be modified to http1.1.

I finally figured it out when error 464 appeared on the screen when I fed the sitemap in google search console.

Measures 

curl http://example.com/ --http1.1
curl http://example.com/ --http2

You will need to check using curl commands at hand, such as

2. Name.com can only register CNAME records for subdomains.

ALB is assigned an IP address, which is not fixed, but changes when it is scaled out, etc.

Therefore, a DNS name is assigned and a CNAME record must be created in the domain's DNS settings.

However, Name.com does not allow the creation of CNAME records except for subdomains. So I thought about transferring my domain to Route53, but that was not possible.

There is a limit to the top domains that can be managed by Route53, and site was not included.

[Top- level domains that can be registered with Amazon Route 53](https://docs.aws.amazon.com/ja_jp/Route53/latest/DeveloperGuide/registrar- tld- list.html)

Therefore, we modified the system by creating a zone in Route53, setting the NS described in the created zone to NS on the Name.com side, and directing the zone from Route53 to ALB.

There is also a way to use the AWS Global Accelerator to assign IP addresses, but I didn't want to spend the money this time & it seems to be quite expensive, so I decided to pass on it.
It seems to improve performance, so I wonder if it would be a good idea to include it if the project is severe in that area.

[AWS Global Accelerator](https://aws.amazon.com/jp/global- accelerator/?blogs- global- accelerator.sort- by=item.additionalFields.createdDate&blogs- global- accelerator.sort- order=desc&aws- global- accelerator- wn.sort- by=item.additionalFields.postDateTime&aws- global- accelerator- wn.sort- order=desc)

When I googled the "unable to reach site" error, all I got was information on the paid version of Hatena Blog (which redirects to www.), and it took me a while to figure out the cause, but I'm glad I solved it.
