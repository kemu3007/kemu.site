サイトの構成について

#### 旧サイト[kemu.site](https://kemu.site) 一般的なEC2を利用したWebサーバの構成になっています。それほどアクセス数を見込んでいないことからpostgresを立てず、sqliteで対応しています。 ロードバランサはALBを利用することでhttp -> httpsのリダイレクト処理を行わせています。CLBを利用した場合この処理ができないため以下のページを参考に行う必要があります。 [Classic Load Balancer で HTTP トラフィックを HTTPS にリダイレクトするにはどうすればよいですか。](https://aws.amazon.com/jp/premiumsupport/knowledge-center/redirect-http-https-elb/) 上記処理を行った場合の欠点として全てのリクエストに対してifの分岐処理が行われるためWebサーバへの負荷が高まりやすいことが挙げられます。また、AWS WAFなども利用することを考えるとALBへ移行した方が良いです。 #### 現行サイト[portal.kemu.site](https://portal.kemu.site) Github Actions上でGithubのissueを収集してjson形式で出力させ、擬似的なバックエンドとして扱うことでdjangoサーバを廃止しました。これにより、Webサーバへのsshセキュリティ、管理手続きが簡略化されています。 また、EC2の料金等に関しても考える必要がないためコスト面での最適化も行われています。 --- 構成図は [tracery](https://tracery.jp/) で記載しました。draw.ioを適当に利用するとgoogle driveのroot周りが汚れていったりしてしまうのでプロジェクトに紐づけて図を作成できるこのツールとても便利。
