关于被Google Adsense、网站结构等卡住的故事。

発行日時: 2022.03.06 05:51

最終更新日: 2022.04.04 05:16

这个网站的结构是

Name.com - > Route 53 - > AWS ALB - （http1.1）> AWS EC2。

该系统的形式如下。刚开始实施的时候，

Name.com - > AWS ALB - (http2.0)> AWS EC2.

但它是在adsense注册的，这是因为ALB的规范而发生的。

在注册adsense的过程中，有两点绊脚石，所以我会把它们记下来。

1.当我试图用adsense注册时，我得到一个错误，说我无法到达该网站

总之，[Http Error 464]（https://docs.aws.amazon.com/ja_jp/elasticloadbalancing/latest/application/load- balancer- troubleshooting.html#http- 464- issues）正在发生。

似乎请求的http和负载均衡器内的协议不匹配，因此出现了错误。

与ALB目标组的通信必须被设置为http1.1，EC2上的nginx也必须被修改为http1.1。

当我在google搜索控制台输入网站地图时，屏幕上出现了错误464，我终于发现了。

###措施。

curl http://example.com/ --http1.1
curl http://example.com/ --http2

你需要使用手头的curl命令进行检查，比如说

2. name.com只能为子域注册CNAME记录。

ALB被分配了一个IP地址，这个地址不是固定的，而是在扩大规模时发生变化，比如说。

因此，要分配一个DNS名称，并且必须在域名的DNS配置中创建一个CNAME记录。

然而，Name.com不允许创建CNAME记录，除非是子域。所以我想过把我的域名转到Route53，因为我觉得Name.com一般很难使用，但这是不可能的。

对Route53可以管理的顶级域名有限制，"site "不在其中。

[可通过亚马逊路线53注册的顶级域名](https://docs.aws.amazon.com/ja_jp/Route53/latest/DeveloperGuide/registrar- tld- list.html)

因此，修改的方法是在Route 53中创建一个区，将创建的区中描述的NS设置为Name.com方面的NS，并将该区从Route 53引导到ALB。

还有一种方法是使用AWS全球加速器来分配和使用IP地址，但我这次不想花钱，而且它似乎相当昂贵，所以我放弃了它。
它似乎可以提高性能，所以如果项目在这方面很严重，我认为包括它是一个好主意。

[AWS全球加速器](https://aws.amazon.com/jp/global- accelerator/?blogs- global- accelerator.sort- by=item.additionalFields.createdDate&blogs- global- accelerator.sort- order=desc&aws- global- accelerator- wn.sort- by=item.additionalFields.postDateTime&aws- global- accelerator- wn.sort- order=desc)

当我在谷歌上搜索 "无法到达网站 "的错误时，我得到的所有信息都是关于Hatena Blog的付费版本（它重定向到www.），我花了一些时间来找出原因，但我很高兴我解决了这个问题。
